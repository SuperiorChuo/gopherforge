# Go Admin Kit 扩展规划

> 基于当前仓库形态（`microservices/` + `monolith/` + React 管理台）整理的中长期扩展蓝图。  
> 原则：**底座通用、业务可插拔、重媒体/重实时独立成仓、管理台统一入口。**

---

## 1. 你现在已经站在哪

| 层 | 现状 | 扩展价值 |
|----|------|----------|
| 产品形态 | 微服务 / 单体双线，业务零调用 | 客户可按规模选型，扩展主要加在微服务线 |
| 身份与权限 | auth + identity + RBAC + 菜单 | 一切新业务的「账号 / 权限 / 租户」地基 |
| 系统能力 | 字典、公告、设置、文件、日志、监控 | 运营后台标配已齐 |
| 实时与事件 | WebSocket 通知雏形、NATS | 可支撑 IM、坐席状态、话单事件 |
| AI | `ai-service` + 对话 / 知识库雏形 | 向 Agent、质检、智能客服演进 |
| 工程 | Compose、网关、CI、OpenAPI | 新服务可按「加目录 + 标签 + 路由」接入 |
| 前端 | React + Ant Design 双线各一份 | 新业务优先做「控制台模块」，少造轮子 |

**一句话定位（建议写进对外叙事）：**

> Go Admin Kit = **企业运营中台底座**（人、权、配、审计、文件、监控）  
> + **可插拔业务域**（AI / 客服 / IM / 工单 / 多端触达）  
> + **外部专业系统对接**（FreeSWITCH 呼叫、第三方 CRM/ERP）

## 2. 扩展总原则

### 2.1 该进本仓 vs 该独立仓

| 类型 | 放哪 | 例子 |
|------|------|------|
| 管理配置、权限、报表、工单状态机 | **本仓**（API + React） | 坐席资料、技能组、质检单、知识库配置 |
| 域业务服务（无重媒体） | **本仓 `microservices/services/*`** | im-service、ticket-service、ai 增强 |
| 媒体 / 信令 / 专用中间件 | **独立仓库** | FreeSWITCH 增强、音视频 SFU、推送网关 |
| C 端触达 | **独立客户端仓** | 微信小程序、H5、App |
| 巨石再次膨胀 | ❌ 禁止 | 别把 IM/FS/AI 全塞回 `monolith/server` |

### 2.2 三条技术红线

1. **新业务优先微服务线**；单体只同步「通用后台能力」，不同步呼叫媒体、全量 IM 等重实时栈。  
2. **统一身份**：所有域服务只认本仓 JWT / 网关 ForwardAuth，禁止各服务自建用户表。  
3. **事件优先**：跨域用 NATS / Webhook（登录、会话结束、话单生成、工单状态），少写同步 RPC 蜘蛛网。

### 2.3 产品分层（目标架构）

```text
┌──────────────────────────────────────────────────────────┐
│  触达层  📱 小程序 / H5 / App / 坐席软电话 / 开放 API      │
└────────────────────────────┬─────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────┐
│  体验台  ⚛️ Go Admin Kit Web（统一 B 端）                 │
│  运营 · 客服 · 质检 · 配置 · 报表 · AI 控制台             │
└────────────────────────────┬─────────────────────────────┘
                             │ Traefik + JWT
┌────────────────────────────▼─────────────────────────────┐
│  业务域服务（本仓 microservices）                         │
│  auth · identity · system · audit · file · monitor       │
│  ai · im · ticket · notify · (cc-adapter) …              │
└───────────────┬─────────────────────────────┬────────────┘
                │                             │
                ▼                             ▼
     PostgreSQL / Redis / NATS      独立专业系统
                                    FreeSWITCH 呼叫仓
                                    短信/邮件网关
                                    大模型 API
```

## 3. 可扩展成的「产品形态」

不必一次做成全部，可按商业目标选 1～2 条主线。

### A. 运营中台（默认主线）✅ 当前最像

- 组织、权限、菜单、字典、公告、文件、审计、监控  
- 适合：内部后台、SaaS 管理端、多业务共用的「权限壳」

### B. 智能客服中台 ⭐ 最推荐的差异化方向

- 会话（IM）+ 机器人（AI）+ 工单 + 质检 + 知识库  
- 坐席工作台在本仓 React；消息与 AI 为独立服务  
- 适合：电商/本地生活/SaaS 客服、企业内部 IT 服务台

### C. 全渠道联络中心（Contact Center）

- B + 呼叫（FreeSWITCH 独立仓）+ 小程序留言 + IM  
- 本仓做「统一客户视图 + 坐席 + 报表」  
- 适合：有电话坐席需求的团队，与纯 IM 客服可分阶段上

### D. AI 运营助手平台

- 强化现有 `ai-service`：知识库、Agent 工具、日志/工单摘要  
- 管理台做模型配置、用量、审计、提示词版本  
- 适合：先卖「提效」再卖「客服」

### E. 多租户 SaaS 脚手架

- 租户、套餐、配额、插件开关（AI/IM/CC 按租户启用）  
- 适合：你要对外卖「可定制运营台」时

### F. 行业套件（后置）

- 在 A～E 稳定后，用「配置 + 插件」做垂直模板（如政务热线、电商客服），而不是硬编码进核心。

## 4. 分阶段路线图（建议 18～24 个月节奏，可压缩）

### Phase 0 · 底座产品化（0～2 个月）

**目标**：开源/交付物「能用、能讲、能发版」。

| 项 | 说明 |
|----|------|
| 发版 | `v0.1.0` + Changelog + 固定 compose 一键体验 |
| 质量 | E2E（登录/RBAC/关键 CRUD）进 CI |
| 文档 | 生产部署（HTTPS、密钥、备份）、二次开发指南 |
| 安全 | 默认密钥检测、管理员强制改密策略开关说清 |
| 观测 | 微服务 metrics / 基础 Grafana 看板模板默认可用 |

**完成标志**：陌生人 clone 后 15 分钟内进后台；CI 绿；有正式 tag。

---

### Phase 1 · 🤖 AI 平台化（2～5 个月）

**目标**：从「能聊」到「能干活、能审计」。

| 模块 | 内容 |
|------|------|
| 模型网关 | 多供应商、多模型、限流、费用计量 |
| 知识库 2.0 | 解析/切片/向量/权限（部门/角色/租户） |
| Agent | 受控工具：查用户、查工单、查日志（全审计） |
| 运营场景 | 公告润色、日志洞察、质检摘要、会话小结 |
| 控制台 | 模型配置、知识库管理、调用审计、用量报表 |

**完成标志**：至少 2 个可演示的 Agent 场景 + 全链路审计。

---

### Phase 2 · 💬 IM + 🎧 智能客服（5～10 个月）

**目标**：形成「客服工作台」差异化。

| 模块 | 内容 |
|------|------|
| `im-service` | 会话、消息、已读、历史、坐席分配 |
| 客服工作台前端 | 排队、转接、标签、快捷回复、侧边客户资料 |
| 人机协同 | 机器人预答 → 一键转人工；转人工带上下文 |
| `ticket-service` | 工单状态机、SLA、与会话绑定 |
| 质检 | 抽样、AI 预质检、人工复核 |

**完成标志**：演示环境可完成「用户进线 → 机器人 → 人工 → 关单 → 质检」。

---

### Phase 3 · 📱 小程序 / 多端触达（可与 Phase 2 后半并行）

**目标**：C 端入口，不替代 B 端后台。

| 模块 | 内容 |
|------|------|
| 小程序仓 | 独立仓库；登录绑定 openId / 手机号 |
| 能力 | 自助查进度、提交工单、在线客服（连 im-service） |
| 触达 | 订阅消息、模板消息（配置在本仓） |
| 原则 | **小程序是通道，运营配置全在 Go Admin Kit** |

**完成标志**：小程序可发起会话并在管理台坐席侧可见。

---

### Phase 4 · ☎️ FreeSWITCH 呼叫中心（独立项目，后置）

**目标**：媒体与管理解耦。

| 仓库 | 职责 |
|------|------|
| **FreeSWITCH 增强项目**（独立） | 拨号计划、IVR、队列、录音、中继、高可用 |
| **本仓** | 坐席/技能组/话单/质检/报表 UI + `cc-adapter` 对接 ESL/HTTP/事件 |

对接事件示例：`call.ringing` / `call.answered` / `call.hangup` / `recording.ready`。

**完成标志**：管理台可见话单与录音回放；坐席状态与 FS 队列一致。

---

### Phase 5 · 🏢 平台化（10～18 个月+）

| 模块 | 内容 |
|------|------|
| 多租户 | 租户隔离、套餐、配额、域名/品牌 |
| 插件市场 | AI / IM / Ticket / CC 按租户开关 |
| 开放平台 | 对外 OpenAPI、Webhook、第三方应用 OAuth |
| 数据看板 | 客服效能、AI 采纳率、渠道转化 |
| i18n | 管理端多语言 |

## 5. 建议新增的服务与仓库清单

### 本仓内（microservices）

| 服务 | 优先级 | 职责 |
|------|:------:|------|
| `ai`（增强） | P0 | 模型、知识库、Agent |
| `im-service` | P0 | 消息与会话 |

| `notify-service` | P1 | 短信/邮件/订阅消息统一出口 |
| `cc-adapter` | P2 | 对接 FreeSWITCH 独立项目 |
| `tenant-service` | P3 | 多租户（若做 SaaS） |

### 独立仓库（推荐）

| 仓库 | 优先级 | 职责 |
|------|:------:|------|
| `*-miniprogram` | P1 | 微信小程序 |
| `freeswitch-cc/`（本 monorepo 顶层目录） | P2 | FreeSWITCH 增强与部署 |
| （可选）`*-mobile` | P3 | App |

## 6. 数据与集成约定（扩展时先定这些）

1. **用户主键**：统一 `user_id`（identity）；外部渠道用 `binding` 表（openId、手机号、SIP 分机号）。  
2. **客户视图**：可引入 `crm-lite`（客户主数据 + 标签），IM/通话/工单都挂 `customer_id`。  
3. **权限码**：新模块沿用 `域:资源:动作`（如 `im:session:transfer`、`cc:recording:play`）。  
4. **菜单种子**：每个域服务自带菜单 migration/seed，或统一由 system 注册中心下发。  
5. **审计**：所有 AI 工具调用、听录音、导话单必须进 audit。  
6. **配置**：模型 Key、中继账号、小程序 appSecret 进安全配置或密钥管理，禁止明文进库前端。

---

## 7. 单体线怎么跟

| 策略 | 说明 |
|------|------|
| **推荐** | 单体 = 轻量运营后台模板；不跟 IM/FS/全量 AI Agent |
| **可选同步** | 工单、基础通知、只读 AI 摘要等「无重实时」能力 |
| **不推荐** | 单体内嵌 FreeSWITCH 或完整 IM 集群逻辑 |

对外话术可以是：

- 微服务版：**中台 / 客服 / AI 全能力**  
- 单体版：**快速后台 / 中小团队 / 内网系统**

## 8. 商业化与开源策略（若你考虑）

| 层级 | 内容 |
|------|------|
| 开源核心 | 认证、RBAC、系统管理、文件、监控、基础 AI、compose |
| 增值/商业（可选） | 多租户、高级质检、呼叫对接套件、行业模板、托管版 |
| 生态 | 插件协议 + 官方 FreeSWITCH 对接文档 + 小程序模板 |

开源项目先把 **Phase 0～2 演示闭环** 做漂亮，比早早写「大而全 roadmap」更重要。

## 9. 风险与克制

| 风险 | 应对 |
|------|------|
| 路线图过大导致核心腐化 | 每阶段只交 1 个可演示闭环 |
| 微服务爆炸 | 新服务必须有边界文档 + 独立 CI job |
| 前端双份漂移 | 中长期可抽 `packages/ui`（别过早） |
| 呼叫/IM 技术深 | 独立仓、专业模块，本仓只做管理与编排 |
| AI 成本与安全 | 计量、限流、脱敏、审计从第一天就做 |

## 10. 近 90 天可执行清单（务实）

若只选「最划算」的下一步：

1. **发 v0.1.0**，README + compose 体验打磨  
2. **AI**：知识库权限 + 1 个 Agent 工具（如查登录日志/工单）  
3. **IM 最小闭环**：会话列表 + 消息 + 管理台坐席页（可不做群聊）  
4. **文档**：二次开发「如何加一个微服务」教程  
5. **FreeSWITCH**：另起仓库写清架构与对接接口草案（先不写进本仓代码）

---

## 11. 总结

Go Admin Kit **最自然的扩展形态**不是「再做一个大而全单体」，而是：

```text
运营中台底座（已有）
    → AI 提效（在做）
    → 智能客服 / IM（高价值）
    → 小程序触达
    → FreeSWITCH 呼叫（独立项目对接）
    → 多租户与开放平台
```

这样既匹配你现在的 **微服务拆分 + 网关 + React 控制台**，又不会把 FreeSWITCH、小程序运行时硬塞进脚手架仓库。
